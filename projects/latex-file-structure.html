LaTeX File Structure

Personal
Figure 1. File structure diagram

For the longest time, I wrote all my LaTeX papers as one giant document that could become incredibly tough to navigate. I spent a long time looking for a file structure template that suits my needs, but was never really satisfied with anything I found. So, I decided to make my own: Link to GitHub project.


One of the things I was really looking for, was something that would allow me to do all the things I've been doing with LaTeX, but without the eternal hassle of copy-pasting document templates. Instead, I wanted something overseeable and simple to operate. Ideally, I was looking for something that resembled the structure used in the Delaunay Mesher project because I found that particularly convenient.


In the past, I have always used Texmaker for LaTeX typesetting, but I have been migrating all my text editing to Atom. Atom provides a few packages for compiling LaTeX in-editor, although I was never really satisfied with those either, as I never really had the level of control I was looking for. Instead, I drew inspiration from the Delaunay Mesher project, and created this structure in such a way that make would do the exact same as build in Texmaker.


So that's where we are. The template structure organizes all the aspects of a LaTeX project (i.e. bibliographies, classes and styles, images, etc.) into a set of sub-folders, which are all brought together into a single .pdf file with the make command. The most immediate consequence of this, is that all sections are split up into their own individual files (with the standalone documentclass), meaning they are much lighter and simpler to navigate than large, bulky projects. Furthermore, re-ordering sections is as simple as moving a single \input{} or \import{}{} line.


Most of the development time went into figuring out the Makefile. I was not too familiar with how to build a Makefile, and had mostly relied on combining functionality from other projects or online examples. A popular solution for my specific problem is the latexmk package, although I was running into too many issues trying to compile these modular documents. Indeed, the simpler solution is usually the better one, so I wrote a Makefile based on some tutorials I had found, and just built my LaTeX documents the same way I had always configured Texmaker: with pdflatex. So that's what it does. It envokes the pdflatex function as many times as I know necessary for most of my previous projects, and makes compiling the bibliography a separate option. Lastly, there are some other options that have made my life easier from time to time, such as clean, verbose for debugging, force in case it's not recognizing changes to sub-files (and it never will), bib and index for making bibliographies and indeces, and superclean to remove pretty much everything and restores the repository to just the bare .tex files.


Now, it has some disadvantages, unfortunately. Primarily, it's not really too efficient as it will always run pdflatex three times, even though that may not be necessary. Furthermore, latexmk has the functionality to watch changes in included .tex files, and would re-compile the document whenever an included file was changed. While useful, this functionality could also be quite infuriating whenever latexmk would get stuck on errors, so it's not included in this project (also in part because I haven't the faintest how I would go about implementing such a thing).
